# ADR-001: 메시지 큐 선택 - Kafka vs RabbitMQ

- **날짜**: 2026-04-03
- **상태**: 결정됨
- **결정자**: 본인

---

## 배경 및 문제 정의

현재 회사에서 Python + RabbitMQ 기반 메시지큐 시스템을 운영하면서 다음 문제를 경험했다.

1. **메시지 유실**: Consumer 프로세스 재시작 시 처리 중이던 메시지가 유실되는 케이스 발생
2. **재처리 불가**: 특정 플랫폼 크롤링 결과를 다시 처리하고 싶어도 메시지가 이미 삭제됨
3. **병렬 처리 한계**: 처리량이 늘어날 때 Consumer를 늘리는 과정이 수동적이고 불안정함

이 포트폴리오 프로젝트에서는 이 문제들을 구조적으로 해결하는 새 설계를 Java/Kotlin으로 구현한다.

---

## 검토한 대안

### Option A: RabbitMQ 유지 (Java 클라이언트로 포팅)
**장점**
- 기존 운영 경험 그대로 활용
- 복잡한 라우팅(Exchange, Binding) 지원
- 설치 및 운영이 Kafka보다 단순

**단점**
- 메시지 소비 후 삭제 → 재처리 불가
- 파티션 기반 병렬성 없음 → 스케일아웃 방식이 단순하지 않음
- 처리량이 증가할수록 성능 한계가 명확 (단일 큐 기준 약 20,000 msg/s)
- 대규모 분산 시스템 환경에서 팬아웃(fan-out) 구조가 필요할 때 Consumer Group이 없어 별도 큐를 여러 개 만들어야 함

### Option B: Apache Kafka (선택)
**장점**
- 메시지 보존 (기본 7일) → 재처리 가능
- 파티션 기반 병렬성 → Consumer를 늘리면 선형적으로 처리량 증가
- Consumer Group으로 독립적 처리 가능 (같은 메시지를 여러 서비스가 소비)
- 네이버/카카오/토스 실제 사용 스택 → 포트폴리오 어필 강함
- Transactional Outbox Pattern 구현에 적합

**단점**
- 운영 복잡도 높음 (Zookeeper 또는 KRaft 필요)
- 단순 RPC-like 패턴에는 오버스펙
- 메시지 순서 보장이 파티션 단위 → 전체 순서 보장은 파티션 1개로 제한

### Option C: Redis Streams
**장점**
- Redis 하나로 캐시 + 메시지큐 통합
- 운영 단순
- Consumer Group 지원

**단점**
- 대용량 처리에서 Kafka 대비 성능 한계
- 메시지 보존 정책이 Kafka만큼 유연하지 않음
- 빅테크 포트폴리오 어필 약함

## AI와 함께 검토한 내용

**내가 AI에게 물어본 것**:
> "현재 프로젝트 규모(일 수십만 건)에서 Kafka를 선택하는 게 오버엔지니어링 아닌가요?"

**AI 답변 요약**:
> "처리량 자체는 RabbitMQ로도 충분하지만, 핵심 질문은 '재처리가 필요한가'와 '여러 Consumer가 같은 데이터를 독립적으로 소비해야 하는가'입니다. 크롤링 결과를 분석 서비스, 알림 서비스, 저장 서비스가 각각 소비해야 한다면 Kafka의 Consumer Group이 구조적으로 훨씬 깔끔합니다."

**내 판단**:
오버엔지니어링 우려는 타당하다. 하지만 이 포트폴리오의 목적이 "실제 규모에서 Kafka를 쓸 판단력이 있는가"를 증명하는 것이므로, Kafka를 선택하고 **왜 선택했는지를 이 ADR로 증명**하는 것이 더 가치 있다.

## 결정: Kafka 선택

### 선택 근거
1. **재처리 요구사항**: 특정 플랫폼 데이터 파싱 로직이 바뀌면 과거 메시지를 다시 처리해야 하는 요구사항이 실제로 있었음
2. **독립적 소비**: 동일 크롤링 결과를 저장(DB), 알림(Slack), 분석(통계) 3개 서비스가 독립적으로 소비 → Consumer Group이 자연스러운 해결책
3. **확장성 증명**: 파티션 수와 Consumer 수의 관계를 코드와 벤치마크로 증명할 수 있음

### 이 결정이 틀렸다고 판단할 기준
- 파티션 수를 초과하는 Consumer를 사용해야 하는 시나리오가 발생한다면 RabbitMQ의 유연한 라우팅이 더 적합할 수 있음
- 메시지 TTL이 7일로 충분하지 않은 장기 보존 요구사항이 생긴다면 별도 저장소가 필요

---

## 구현 결정 사항

| 설정 | 값 | 근거 |
|-----|---|-----|
| 파티션 수 | 6 | Consumer 최대 6개 기준, 6의 배수로 설정 |
| Replication Factor | 1 (로컬), 3 (프로덕션 가정) | 로컬은 단일 브로커 |
| `enable.auto.commit` | false | Manual Commit으로 메시지 처리 보장 |
| `max.poll.records` | 100 | 단일 poll에서 처리하는 메시지 수 (처리 시간 * 100 < max.poll.interval.ms) |
| `max.poll.interval.ms` | 300,000ms (5분) | 기본값 유지, 처리 로직이 5분 이내 완료 보장 |
